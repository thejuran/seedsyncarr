<objective>
Make `MultiprocessingLogger` work correctly under BOTH `fork` (Linux default) and `spawn`
(macOS default) multiprocessing start methods by creating its internal queue from a shared
`spawn`-compatible multiprocessing context, and update the three analog tests to deterministically
exercise the spawn path on every platform (including Linux CI). This delivers INFRA-01.

Purpose: A bare `multiprocessing.Queue(-1)` created in the default (fork) context produces a
`SemLock` with `_is_fork_ctx=True`. When that queue is shared with a `spawn`-started child (macOS
default, or any spawn-context `Process`), CPython's `SemLock.__getstate__` raises
`RuntimeError: A SemLock created in a fork context is being shared with a process in a spawn context`.
A `spawn`-context queue's SemLocks have `_is_fork_ctx=False` and are safely shareable with BOTH
fork and spawn children — fixing macOS without regressing fork-default Linux. This also implicitly
fixes production correctness for `AppProcess` (a `multiprocessing.Process` subclass) on macOS.

Output:
- `src/python/common/multiprocessing_logger.py` — queue created from a stored spawn context; the
  context exposed so children can be launched from the same context.
- `src/python/tests/unittests/test_common/test_multiprocessing_logger.py` — the three `process_1`
  closures promoted to module-level picklable targets; child `Process` launched via the logger's
  spawn context. Assertions, `LogCapture`, `@pytest.mark.timeout`, and bounded `join` unchanged.
</objective>
