<objective>
Make `MultiprocessingLogger` work correctly under BOTH `fork` (Linux default) and `spawn`
(macOS default) multiprocessing start methods, and — critically — make a `MultiprocessingLogger`
INSTANCE survive being pickled by `spawn`. The fix has two coupled parts in one production module:

1. Create the internal queue from a shared `spawn`-compatible multiprocessing context (D-01/D-02),
   so the queue's `SemLock` is shareable with both fork and spawn children.
2. Add `__getstate__`/`__setstate__` to `MultiprocessingLogger` so that when `spawn` pickles the
   instance (which the three analog tests do via `args=(mp_logger,)`, and which production
   `AppProcess.start()` does via `self.mp_logger`), only the child-side picklable state is
   serialized — the spawn-context queue, the logger level, and the base `logging.Logger` — while
   the main-process-only `threading.Thread` listener, the `threading.Event` shutdown flag, and the
   transient `sys.exc_info()` triple are EXCLUDED.

Then update the three analog tests to deterministically exercise the spawn path on every platform
(including Linux CI) by promoting their closures to module-level picklable targets and launching
children from the logger's spawn context, asserting each child exits cleanly. This delivers INFRA-01.

Purpose: The original (queue-only) plan fixed the FIRST serialization failure but not the whole
bug. The three analog tests launch their child via `args=(mp_logger,)`, so `Process.start()` under
spawn PICKLES the entire `MultiprocessingLogger`. That instance holds a `threading.Thread`
(multiprocessing_logger.py:26-27) and a `threading.Event` (line 28), both backed by `_thread.lock`,
which is unpicklable — so `start()` raises `TypeError: cannot pickle '_thread.lock' object` BEFORE
the child runs, even after the queue's `SemLock` is fixed. This is NOT merely a test concern: the
production subclass `AppProcess` (common/app_process.py) extends `multiprocessing.Process`, stores
`self.mp_logger` (lines 45, 50-51), and calls `self.mp_logger.get_process_safe_logger()` in `run()`
(lines 74-75). Under spawn, `AppProcess.start()` pickles the whole instance including `self.mp_logger`
and hits the same `_thread.lock` wall. A genuine INFRA-01 fix must make `MultiprocessingLogger`
spawn-picklable, not just fix the queue. The child never starts a listener — it only PUTs records
onto the queue via `get_process_safe_logger()`'s `QueueHandler` — so the child needs only the
queue + level (+ base logger), which `__getstate__`/`__setstate__` provide.

Output:
- `src/python/common/multiprocessing_logger.py` — queue created from a stored spawn context; the
  context exposed; `__getstate__`/`__setstate__` make the instance spawn-picklable.
- `src/python/tests/unittests/test_common/test_multiprocessing_logger.py` — the three `process_1`
  closures promoted to module-level picklable targets; child `Process` launched via the logger's
  spawn context; each child asserted `exitcode == 0`. All other assertions, `LogCapture`,
  `@pytest.mark.timeout`, and bounded `join` unchanged.
</objective>

CRITICAL name-mangling fact: `self.__listener` (the attribute, L26) is assigned a Thread whose
`target=self.__listener` is the BOUND METHOD of the same mangled name. After __init__ runs, the
INSTANCE attribute `_MultiprocessingLogger__listener` is the Thread object (it shadows the method on
the instance; the method remains reachable on the CLASS). So instance __dict__ after construction
holds these mangled keys:
  - "logger"                                  (logging.Logger)            KEEP for child
  - "_MultiprocessingLogger__mp_context"      (spawn context)            KEEP for child (added by this plan)
  - "_MultiprocessingLogger__queue"           (spawn-ctx Queue)          KEEP for child
  - "_MultiprocessingLogger__logger_level"    (int)                      KEEP for child
  - "_MultiprocessingLogger__listener"        (threading.Thread)         DROP — unpicklable, main-only
  - "_MultiprocessingLogger__listener_shutdown" (threading.Event)        DROP — unpicklable, main-only
  - "_MultiprocessingLogger__listener_exc_info"  (None or exc triple)    DROP — transient, main-only

What the CHILD actually needs after unpickle: get_process_safe_logger() (L48-65) reads ONLY
self.__queue (builds QueueHandler from it) and self.__logger_level (root.setLevel). It does NOT
touch the listener, the event, or exc_info. The child must NEVER call start()/stop(). So on
__setstate__ the dropped fields may be restored to None (or left unset); the child never references
them. (Restoring them to None is safer than leaving them unset so an accidental attribute read
fails loudly with AttributeError-free None rather than a confusing missing-attr — Claude's
Discretion, but be explicit and consistent.)
