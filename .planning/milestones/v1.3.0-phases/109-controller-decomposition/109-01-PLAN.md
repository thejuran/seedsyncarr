<objective>
Extract the command-dispatch handler logic out of the `Controller` god-class into a new
`controller/command_processor.py` collaborator (the most self-contained seam, clearest public-contract
boundary via `queue_command`). This is plan 1 of 3 in the behavior-preserving Controller decomposition
(ARCH-01, D-06 — lowest-risk seam first).

The four `__handle_{queue,stop,extract,delete}_command` methods (`controller.py:1009-1097`) move into a
`CommandProcessor` class as single-underscore `_handle_*` methods behind a public `handle(file, command)`
router. The `__process_commands` drain/lock/dispatch scaffold (`controller.py:1099-1141`) STAYS on
`Controller` (tests call `_Controller__process_commands()` directly) and delegates to the collaborator
after the model lock is released. The `Command` inner class STAYS nested on `Controller` (it is referenced
as `Controller.Command` in `__process_commands`'s `_notify_failure` closure and as a type annotation).

TEST-PINNED-HELPER GUARD (structural rule shared by all three 109 plans): the four moving methods are
DUNDER (`__handle_*`), so they name-mangle to `_Controller__handle_*` and the suite never calls them
directly — deleting them is safe. The methods the suite DOES call directly on a Controller instance
(`_Controller__process_commands`, plus `queue_command`) STAY on Controller. Before deleting any method
definition, the executor greps `src/python/tests/` for direct `controller.<name>(` / `self.controller.<name>(`
call sites; any name with a hit keeps a callable def (a forwarding wrapper if its body moves) — only the
body relocates, never the callable name.

Purpose: maintainability — shrink the god-class without changing one observable behavior (ARCH-01).
Output: `controller/command_processor.py` (new), thinned `controller.py`, `__init__.py` export added.
</objective>

The four handle methods to extract (controller.py:1009-1097), each NO-lock, each returning a
(success, error_msg, error_code) triple:
  __handle_queue_command(self, file: ModelFile, command: Command) -> (bool, str, int)   # :1009-1022
  __handle_stop_command(self, file: ModelFile, command: Command) -> (bool, str, int)    # :1024-1037
  __handle_extract_command(self, file: ModelFile, command: Command) -> (bool, str, int) # :1039-1057
  __handle_delete_command(self, file: ModelFile, command: Command) -> (bool, str, int)  # :1059-1097

These depend ONLY on: self.__lftp_manager, self.__file_op_manager, self.__persist, self.logger.
None acquire __model_lock. __handle_delete_command calls self.__file_op_manager.delete_local(file)
which spawns a subprocess — it MUST run with no lock held (already true; __process_commands releases
__model_lock before calling any handler).

These four are DUNDER (mangled to _Controller__handle_*) and the test suite never calls them directly —
confirmed: `grep -rnE 'controller\._handle_(queue|stop|extract|delete)' src/python/tests/` returns nothing.
So they may be deleted from Controller with no forwarding wrapper.

Command shape (duck-typed inside the collaborator — do NOT import Controller.Command):
  command.action   -> Controller.Command.Action enum member (compare via .name: 'QUEUE'/'STOP'/'EXTRACT'/'DELETE_LOCAL'/'DELETE_REMOTE')
  command.filename -> str
  command.callbacks -> list

The scaffold that STAYS on Controller (controller.py:1099-1141):
  __process_commands(self)  — drains __command_queue, looks up file under __model_lock, RELEASES the lock,
                              then dispatches. Tests call c._Controller__process_commands() at
                              test_controller.py:279,311.
  Its nested _notify_failure(_command: Controller.Command, _msg, _code=400) closure references
  Controller.Command as a type annotation — so the Command inner class stays nested on Controller.

Construction site (controller.py:95-198) — managers constructed here, must STAY here (D-05):
  self.__lftp_manager, self.__file_op_manager are already constructed in __init__ before the new
  collaborator. Inject them (already-constructed) into CommandProcessor.
