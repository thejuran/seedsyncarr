<objective>
Extract the auto-delete BFS pack-guard + coverage-check logic out of `Controller` into a new
`controller/auto_delete_manager.py` collaborator. This is plan 2 of 3 (D-06 — the thread-safety-critical
seam, taken after command dispatch is already out so it can be isolated).

**This is the highest-risk plan.** The two dunder lifecycle methods — `__schedule_auto_delete`
(`controller.py:823-838`) and `__execute_auto_delete` (`controller.py:840-1007`) — MUST stay defined on
`Controller`: tests call `self.controller._Controller__execute_auto_delete(...)` and
`_Controller__schedule_auto_delete(...)` in 30+ locations (109-RESEARCH.md §"Complete Field Inventory"),
and the `threading.Timer` callback must remain a bound `Controller` method. `__execute_auto_delete` stays
as the **lock-management harness** (it owns the `with __auto_delete_lock` / `with __model_lock` /
re-acquire-`__auto_delete_lock` sequence and the WR-02 final commit + the outside-all-locks
`delete_local` dispatch). Only the inner **BFS traversal + coverage decision** (the loop body inside the
`with __model_lock:` window, plus the `_VIDEO_EXTENSIONS` / `_AUTO_DELETE_BFS_NODE_LIMIT` constants) moves
into `AutoDeleteManager.run_bfs_and_coverage(...)`, which acquires NO lock (the caller holds `__model_lock`).

Purpose: maintainability — isolate the auto-delete decision logic with zero change to thread-safety
behavior (ARCH-01 criterion #4). Output: `controller/auto_delete_manager.py` (new), thinned
`__execute_auto_delete`, `__init__.py` export added.
</objective>
