<objective>
Fix BUG-03 (criterion #2): an auto-delete callback fired via threading.Timer can run a deletion
against a half-torn-down controller during shutdown. The timer TRACKING + cancel-on-exit half of
BUG-03 (criterion #1) ALREADY shipped in Phase 101 Plan 05 (self.__pending_auto_deletes dict +
self.__auto_delete_lock + exit()'s cancel/clear loop) — this plan does NOT re-implement it; it adds
a test for it (verify-only per D-01) and closes criterion #2.

Purpose: Guarantee "shutdown begins before the delete is committed ⇒ no delete and no persist-state
mutation", with NO preempt-after-check race. The naive single-entry guard is insufficient: the
callback acquires __model_lock and does a model read + BFS (controller.py:859-951) BEFORE dispatching
delete_local OUTSIDE the lock (955). A callback that passes an entry check (event unset), then blocks
during the model-lock work while exit() sets the event + stops __file_op_manager, would still pop
imported_children (951) and call delete_local (955) against a stopped subsystem. Fix: a dedicated
threading.Event set UNDER __auto_delete_lock in exit(), and a final commit step (shutdown-check →
imported_children.pop → capture file) serialized UNDER __auto_delete_lock in the callback, with
delete_local dispatched only after releasing that lock. Because exit() sets the event under the same
lock, set() cannot interleave between the final check and the commit — strict serialization (D-02,
D-04). No thread join/drain (D-04).
Output: A dedicated shutdown Event with an entry guard + a lock-serialized final-commit guard, plus a
synchronous regression test (entry, mid-callback, positive control) that asserts no delete AND no
imported_children mutation when suppressed.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Timer thread → controller state during shutdown | An auto-delete Timer callback reads the model and dispatches a destructive `delete_local` (subprocess). During `exit()` the read-to-dispatch span crosses lock acquire/release windows where shutdown can interleave; an unguarded dispatch deletes user data against a half-torn-down subsystem. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-102-01 | Tampering / data-loss (destructive op vs torn-down state) | __execute_auto_delete during shutdown | mitigate | BUG-03 itself: a dedicated threading.Event set UNDER __auto_delete_lock in exit(), checked at callback entry and again in a final-commit block serialized on the SAME lock (final shutdown-check → imported_children.pop → capture file; delete_local dispatched after releasing the lock). Strict "shutdown-before-commit ⇒ no commit" — no preempt-after-check race, no persist-state mutation on suppression. |
| T-102-LOG | Information disclosure (log injection) | new/changed log lines in the auto-delete path | mitigate | Any log line interpolating file_name routes through the existing sanitize_log_value() helper (already used throughout this method) — CWE-117 consistent with the file's convention. The shutdown early-returns add no raw-string log lines. |
| T-102-SC | Tampering | dependencies | accept | No new packages; threading.Event and unittest.mock are stdlib, already present. No install task — supply-chain surface unchanged. |
</threat_model>
