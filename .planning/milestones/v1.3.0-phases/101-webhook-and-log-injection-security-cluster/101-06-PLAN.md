<objective>
Close the remaining SEC-01 log-injection sinks in the **lftp subsystem** and the **remote-scanner raw-output** site — the job-name and raw-lftp-output log sites in `lftp/lftp.py`, the parse-error log sites in `lftp/job_status_parser.py`, and the JSON-decode-error log in `controller/scan/remote_scanner.py`. These interpolate values derived from the seedbox remote directory listing / torrent names / raw SSH scan output (a crafted remote filename appears verbatim in these log lines), so a CR/LF/control byte in a remote name can forge or split a log entry (CWE-117). Adversarial-review round 2 confirmed the lftp sites are taint-reachable; round-3 Option-C folds in the highest-risk raw-remote-output site (remote_scanner.py:118).

This plan owns the `lftp/*` files PLUS `controller/scan/remote_scanner.py` — all disjoint from Plan 04 (controller/webhook_manager) and Plan 05 (controller/model). remote_scanner.py is a NEW file for the phase and does NOT collide with any same-wave plan: Plan 04 (Wave 2) touches controller.py/webhook_manager.py, never remote_scanner.py or lftp/*. It depends only on the Plan 01 helper, so it runs in Wave 2 in parallel with Plan 04 (zero file overlap).

- job_status_parser.py:724 ("LftpJobStateParser error: {str(e)}" — the parser ValueError messages interpolate raw remote line/name content at 432/514/552/579/593).
- remote_scanner.py:118 ("JSON decode error: {}\n{}".format(str(err), out)) — ROUND-3 Option-C addition. `out` is the raw SSH scan command output (remote-produced; crafted remote file names can ride in it); `str(err)` is a JSON parse error that can also echo raw input fragments. Sanitize BOTH interpolated values.

Excluded WITH per-site taint rationale (do NOT sanitize):
- lftp.py:114 ("command: {}".format(command.encode('utf8','surrogateescape'))) — the value is a Python `bytes` object whose `repr` already escapes CR/LF and control bytes to `\r`/`\n`/`\xHH`; no raw control byte can reach the log here, so sanitizing is redundant and would double-escape the visible text. Verified: a `\r\n`-bearing name in the command produces no raw newline in the logged bytes repr.
- lftp.py:119, 139 ("Lftp timeout exception") — a hardcoded constant with no interpolation; not taint-reachable.
- lftp.py:309 (f"Ignoring status error (count={self.__consecutive_status_errors})") — interpolates an integer counter, not a remote string; not taint-reachable.

ACCEPTED TRADEOFF (job_status_parser.py:725 and remote_scanner.py:118 `out`): the logged value is the full multi-line raw output block, so sanitizing it escapes its legitimate internal newlines too (the diagnostic block renders as one line with `\n` tokens). This is CORRECT and intended for CWE-117 — the whole point is that NO attacker-controlled newline can forge a log line. This is the documented default per the revision directive; it is not a bug.

NAMEERROR SAFETY (remote_scanner.py:118 — MANDATORY): `out_str` is defined at line 114 (`out_str = out.decode('utf-8') if isinstance(out, bytes) else out`) INSIDE the `try`. The except at line 117 catches the very `ValueError`/`TypeError`/`KeyError`/`json.JSONDecodeError` that the decode-or-json path can raise — so `out_str` may be UNDEFINED in the except branch (referencing it would raise NameError and crash the scan path). The action below specifies deriving a safe decoded string locally inside the except (decode `out` with `errors='replace'` when bytes, else `str(out)`) and sanitizing THAT — never referencing `out_str` in the except.

This is strict hardening, log-output only. The raw `name` is still used for matching and the kill dispatch, the raw `out` is still returned from `__run_command` and consumed by callers/`__detect_errors_from_output`, the raw `output` is still parsed by `parse()`, and the raw `out` in remote_scanner is unchanged (only the LOG line in the except is sanitized; the raised ScannerError and control flow are untouched). Only the interpolated LOG variable is sanitized (D-01/D-02).

Output: 10 lftp log sites + 1 remote_scanner site (2 interpolated values) routed through sanitize_log_value (3 files); injection tests pinning escaped log output and the unchanged raw return/parse values; 3 lftp sites excluded with recorded rationale.
