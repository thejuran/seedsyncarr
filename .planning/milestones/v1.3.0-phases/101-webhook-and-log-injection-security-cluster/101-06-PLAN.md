<objective>
Close the remaining SEC-01 log-injection sinks in the **lftp subsystem** — the job-name and raw-lftp-output log sites in `lftp/lftp.py` and the parse-error log sites in `lftp/job_status_parser.py`. These interpolate values derived from the seedbox remote directory listing / torrent names (a crafted remote filename appears verbatim in these log lines), so a CR/LF/control byte in a remote name can forge or split a log entry (CWE-117). Adversarial-review round 2 confirmed these are taint-reachable and the prior blanket "out of scope this slice" deferral had no sound per-site rationale.

This plan owns ONLY `lftp/*` files — disjoint from Plan 04 (controller/webhook_manager) and Plan 05 (controller/model). It depends only on the Plan 01 helper, so it runs in Wave 2 in parallel with Plan 04 (no file overlap: lftp/* vs controller.py/webhook_manager.py).

Site set for THIS plan (taint-reachable — sanitize):
- lftp.py:356 (kill: "Kill failed to find job '{name}'") — name is the remote-derived job/torrent name.
- lftp.py:362 (kill: "Killing running job '{name}'...") — same.
- lftp.py:365 (kill: "Killing queued job '{name}'...") — same.
- lftp.py:126 ("out ({} bytes):\n {}" — the raw lftp process output, embeds remote file/job names; only edge-stripped, internal CR/LF survive).
- lftp.py:129 ("after: {}" — the raw lftp `after` match buffer, same raw-output taint class).
- lftp.py:144 ("retry out ({} bytes):\n {}" — raw retry output).
- lftp.py:147 ("retry after: {}" — raw retry after buffer).
- lftp.py:148 ("Lftp detected error: {error_out}" — raw lftp output, logged unconditionally on error).
- job_status_parser.py:725 ("Status:\n{output}" — the FULL raw `jobs -v` block; multi-line by nature — see accepted-tradeoff note below).
- job_status_parser.py:724 ("LftpJobStateParser error: {str(e)}" — the parser ValueError messages interpolate raw remote line/name content at 432/514/552/579/593).

Excluded WITH per-site taint rationale (do NOT sanitize):
- lftp.py:114 ("command: {}".format(command.encode('utf8','surrogateescape'))) — the value is a Python `bytes` object whose `repr` already escapes CR/LF and control bytes to `\r`/`\n`/`\xHH`; no raw control byte can reach the log here, so sanitizing is redundant and would double-escape the visible text. Verified: a `\r\n`-bearing name in the command produces no raw newline in the logged bytes repr.
- lftp.py:119, 139 ("Lftp timeout exception") — a hardcoded constant with no interpolation; not taint-reachable.
- lftp.py:309 (f"Ignoring status error (count={self.__consecutive_status_errors})") — interpolates an integer counter, not a remote string; not taint-reachable.

ACCEPTED TRADEOFF (job_status_parser.py:725): the logged value is the full multi-line raw `output` block, so sanitizing it escapes its legitimate internal newlines too (the diagnostic block renders as one line with `\n` tokens). This is CORRECT and intended for CWE-117 — the whole point is that NO attacker-controlled newline can forge a log line. This is the documented default per the revision directive; it is not a bug. (Per-line sanitization was considered and rejected: a crafted remote name can span what looks like a "line", so escaping the whole block is the safe choice.)

This is strict hardening, log-output only. The raw `name` is still used for matching and the kill dispatch (`status.name == name`, `self.__run_command("kill {id}")`), the raw `out` is still returned from `__run_command` and consumed by callers/`__detect_errors_from_output`, and the raw `output` is still parsed by `parse()` — only the interpolated LOG variable is sanitized (D-01/D-02).

Purpose: Complete SEC-01 taint coverage across the lftp subsystem so no remote-derived job name or raw lftp output reaches a log line unescaped.
Output: 10 lftp log sites routed through sanitize_log_value (2 files); injection tests pinning escaped log output and the unchanged raw return/parse values; 3 sites excluded with recorded rationale.
