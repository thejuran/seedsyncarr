<objective>
Harden the webhook endpoint with two additive, backward-compatible changes:

- **BUG-02 (highest priority):** New opt-in flag `general.webhook_require_secret` (bool, default `false`). When ON and no secret is configured, `_verify_hmac` returns 503 BEFORE the body is read. When OFF (default), the locked `Empty webhook_secret skips HMAC` contract is preserved byte-for-byte.
- **SEC-03:** Wrap both webhook routes with the existing `rate_limit(60, 60.0)` decorator (independent per-route closures → sonarr and radarr do not share a budget; over-limit → 429).

COMPAT is the hardest constraint here. Existing config files MUST load unchanged, and existing webhook behavior with the flag off MUST be identical. The default-off flag exists specifically to preserve the `Empty webhook_secret skips HMAC` decision (RESEARCH.md Pitfall 1).

Purpose: An operator can fail-closed the webhook on demand without breaking any existing install on upgrade, and the endpoint is DoS-protected like other mutable routes.
Output: config flag wired via the proven 3-step PROP pattern; `_verify_hmac` 503 branch; rate-limited routes; extended startup warning; full test coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| *arr webhook POST → webhook handler | Untrusted remote caller; without a secret the endpoint authenticates nobody |
| Anonymous network client → webhook route | Unbounded request rate can exhaust the controller import pipeline (DoS) |
| Old on-disk config → Config.from_dict | Upgrade path: existing config files must keep loading |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-101-03 | Spoofing | webhook.py _verify_hmac (unauthenticated acceptance, BUG-02) | mitigate | When operator opts in (webhook_require_secret=True) and no secret is set, return 503 before body parse — fail closed. Default off preserves the documented contract for existing installs. |
| T-101-04 | Denial of Service | webhook.py add_routes (unbounded call rate, SEC-03) | mitigate | rate_limit(60, 60.0) per route → 429 over limit; independent closures so one route cannot exhaust the other's budget. 60/60s is well above legitimate *arr callback frequency. |
| T-101-05 | DoS / Tampering (INTRODUCED) | BUG-02 mis-gating breaks legitimate webhooks on upgrade | mitigate | Default `false` + from_dict back-compat fallback means existing config files load unchanged and flag-off path is byte-identical (TestWebhookIntegration still green; explicit test_require_secret_off_no_secret_returns_200). Pitfall 1 guard: the require_secret-false branch returns None unconditionally. |
| T-101-06 | DoS (INTRODUCED) | SEC-03 rate-limit blocks legitimate bursty callbacks | accept | 60/60s per route matches config/status endpoints; an *arr instance fires at most once per import event, far below the limit. Independent per-route budgets prevent cross-route starvation. Low severity, accepted. |
| T-101-07 | Tampering (INTRODUCED) | from_dict injects Python False instead of "False" string | mitigate | Inject the STRING "False" (Pitfall 3) so Converters.bool/_strtobool runs; test_general_webhook_require_secret_back_compat pins default-False load. |
| T-101-SC | Tampering | npm/pip/cargo installs | accept | No new packages (RESEARCH.md Package Legitimacy Audit: zero new deps). Gate not triggered. |
</threat_model>
