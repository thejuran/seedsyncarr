<objective>
Harden the webhook endpoint with two additive, backward-compatible changes:

- **BUG-02 (highest priority):** New opt-in flag `general.webhook_require_secret` (bool, default `false`). When ON and no secret is configured, the webhook is rejected with 503 BEFORE the body is read AND **before the rate-limit counter is consulted** (so an exhausted rate-limit window cannot mask the fail-closed 503 with a 429). When OFF (default), the locked `Empty webhook_secret skips HMAC` contract is preserved byte-for-byte.
- **SEC-03:** Wrap both webhook routes with the existing `rate_limit(60, 60.0)` decorator (independent per-route closures -> sonarr and radarr do not share a budget; over-limit -> 429).

**Ordering (BLOCKER 2 fix):** the `rate_limit` decorator returns 429 BEFORE calling its wrapped handler, and `_verify_hmac` only runs INSIDE `_handle_webhook` (i.e. inside the rate-limited callable). Putting the 503 inside `_verify_hmac` therefore lets a 429 mask the required 503 once the window is exhausted. The fix is a thin **fail-closed guard wrapper applied OUTSIDE `rate_limit`**: each route is registered as `fail_closed_guard(rate_limit(60, 60.0)(handler))`. The guard reads no body, consults no counter, and short-circuits to 503 when `webhook_require_secret` is on and no secret is set — so 503 always precedes both the 429 branch and any body read. The per-route `rate_limit` closures stay independent (D-09).

COMPAT is the hardest constraint. Existing config files MUST load unchanged, newly-generated default configs MUST load on the next boot, and existing webhook behavior with the flag off MUST be identical. The default-off flag exists specifically to preserve the `Empty webhook_secret skips HMAC` decision (RESEARCH.md Pitfall 1).

Purpose: An operator can fail-closed the webhook on demand without breaking any existing install on upgrade or any fresh install on first run, and the endpoint is DoS-protected like other mutable routes.
Output: config flag wired via the proven 3-step PROP pattern + first-run default; outer fail-closed guard ahead of rate_limit; rate-limited routes; extended startup warning; full test coverage including the 503-before-body-read and 503-survives-rate-limit proofs.
</objective>
