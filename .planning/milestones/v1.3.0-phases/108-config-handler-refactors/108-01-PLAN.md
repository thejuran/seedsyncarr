<objective>
Make `Config` secret-field discovery fully declarative (ARCH-02). Each secret field's `PROP` declaration carries `secret=True`; the encrypt/decrypt code paths discover secret fields dynamically from property metadata instead of from the hand-maintained `_SECRET_FIELD_PATHS` tuple, which is removed entirely. Adding a future secret field requires only declaring it `secret=True` — nothing else.

Purpose: Eliminates the "declare in two places" duplication flagged in CONCERNS.md ("Duplicate-by-five secret-field walk"). Removes a class of silent-drift bug where a new secret field is encrypted but not redacted (or vice versa) because someone forgot to update the tuple.

Output: Updated `config.py` (extended `PropMetadata`, threaded `secret` kwarg, new discovery API, repointed encrypt/decrypt loops, tuple deleted); repointed `seedsyncarr.py` startup hook; new auto-discovery test in `test_config.py` with the existing Fernet suite untouched.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk config file → in-memory Config | Plaintext or Fernet-ciphertext secret values cross into the app on load/decrypt |
| in-memory Config → on-disk config file | Secret values cross out on save/encrypt; correct discovery decides what is encrypted-at-rest |
| developer declaration → encrypt/decrypt loops | The `secret=True` marker is now the SOLE source of truth for which fields are encrypted/redacted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-108-01-01 | Information Disclosure | discovery API silently drops a secret field | mitigate | New test asserts discovery returns exactly the five known triples; existing Fernet round-trip suite (covers all five) stays green — a dropped field would leave its value plaintext on disk and fail the round-trip test. |
| T-108-01-02 | Tampering / Denial of Service | a non-secret field wrongly flagged secret (over-encryption → load failure) | mitigate | `grep` asserts exactly 5 `secret=True` declarations and they are the known five; from_dict/from_str of a normal config still succeeds in the existing suite (an over-encrypted non-secret field would corrupt on next load). |
| T-108-01-03 | Information Disclosure | Encryption.enabled flag becomes discoverable as secret → enabled value encrypted → encryption can never be read back | mitigate | Encryption.enabled keeps secret=False default; a dedicated new test asserts it is absent from discovery output (T-81-02-07 preserved). |
| T-108-01-04 | Tampering | iterator swap changes loop body / on-disk format | accept-after-verify | Loop bodies kept byte-identical (iterator-only swap); no on-disk format change; regression suite is the proof. Low risk — pure refactor. |
| T-108-01-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan (uses existing stdlib + already-vendored deps); RESEARCH Package Legitimacy Gate not triggered. |
