<objective>
Make `Config` secret-field discovery fully declarative (ARCH-02). Each secret field's `PROP` declaration carries `secret=True`; the encrypt/decrypt code paths discover secret fields dynamically from property metadata instead of from the hand-maintained `_SECRET_FIELD_PATHS` tuple, which is removed entirely. Adding a future secret field requires only declaring it `secret=True` — nothing else.

OUT OF SCOPE — the API-response redaction path (`src/python/web/serialize/serialize_config.py`, `_SENSITIVE_FIELDS`) is intentionally NOT touched (ROADMAP scope amendment 2026-06-01; criteria #1/#2 amended). `_SENSITIVE_FIELDS` is an independent, intentional SUPERSET of the 5-field encrypt set — it also redacts the non-secret Lftp connection fields `remote_address`/`remote_username`/`remote_path`. Redaction (info-disclosure prevention in the API response) and encryption-at-rest are deliberately separate concerns: redaction ≠ encryption-at-rest. Coupling them to the `secret=True` metadata would either drop those 3 from redaction (info-disclosure regression) or wrongly encrypt them. ARCH-02 covers ENCRYPT + DECRYPT declarative discovery only. `_SENSITIVE_FIELDS` stays byte-for-byte as-is.

Purpose: Eliminates the "declare in two places" duplication flagged in CONCERNS.md ("Duplicate-by-five secret-field walk"). Removes a class of silent-drift bug where a new secret field is encrypted but not decrypted because someone forgot to update the tuple.

Output: Updated `config.py` (extended `PropMetadata`, threaded `secret` kwarg, new discovery API, repointed encrypt/decrypt loops, tuple deleted); repointed `seedsyncarr.py` startup hook; new auto-discovery + Fernet-round-trip test in `test_config.py` with the existing Fernet suite untouched. `serialize_config.py` / `_SENSITIVE_FIELDS` is NOT modified.
</objective>

| on-disk config file → in-memory Config | Plaintext or Fernet-ciphertext secret values cross into the app on load/decrypt |
