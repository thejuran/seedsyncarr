# Requirements: SeedSyncarr

**Defined:** 2026-06-02
**Milestone:** v1.4.0 — Launch-Hardening for Public Release
**Core Value:** Reliable file sync from seedbox to local with automated media library integration

## v1.4.0 Requirements

Requirements for the launch-hardening milestone. Each maps to exactly one roadmap phase.
Source: the approved design spec (brainstormed 2026-06-02). Decisions D-1..D-10 govern scope.

### Hostile-Reader Audit (SCAN)

A bounded "what would a skeptical engineer on Reddit find" discovery pass. Gates the fix scope.
Only genuinely high-visibility findings fold into later phases; everything else is parked with
written rationale (D-3).

- [x] **SCAN-01**: A maintainer can read a triaged findings artifact that lists, by severity, what a skeptical engineer reviewing the public repo would flag — produced by reading the entry points, running the project's existing tooling (ruff whole-tree, Semgrep/Shield, dependency audit) under launch framing, and skimming the highest-traffic source files.
- [x] **SCAN-02**: Each finding in the artifact is marked either "fold into a v1.4.0 fix phase" (with the target phase) or "parked" (with a one-line rationale), so scope decisions are explicit and traceable rather than implicit.

### Config-Set Endpoint Migration (CFG)

The one breaking HTTP-contract change: credentials must stop traveling in URLs/logs. Hard cutover
(POST only; GET path deleted) per D-4. On-disk persisted config format is unchanged.

- [ ] **CFG-01**: A client sets a config value by sending a POST request with a JSON body (`{section, key, value}`) to the config-set endpoint; credential values no longer appear as URL path segments in server access logs, browser history, or reverse-proxy logs.
- [ ] **CFG-02**: The legacy `GET /server/config/set/{section}/{key}/{value}` path no longer exists — a request to it returns a not-found/method-not-allowed response (the credential-leaking path is fully removed, not deprecated-but-live).
- [ ] **CFG-03**: The Settings page saves configuration successfully end-to-end against the new POST endpoint — the Angular `ConfigService` and the E2E setup script/page objects use POST, and a saved setting round-trips and persists across reload.
- [ ] **CFG-04**: Existing on-disk config files (plaintext and Fernet-encrypted) load unchanged after the migration — there is no config-format change and no user migration step for saved settings.

### Defensive Guards & Code Hardening (GUARD)

Close the remaining "a hostile reader will notice this" code items: insecure-by-silence defaults,
a silently-swallowed delete failure, a red test, and two cheap repo-hygiene tells. Defaults stay
backward-compatible (no breaking behavior change) — the app simply stops being *quietly* unsafe.

- [ ] **GUARD-01**: When the server binds to a non-loopback interface with no `api_token` configured, the operator sees a prominent startup warning that API endpoints are unauthenticated — the default behavior is unchanged, but the unsafe posture is no longer silent.
- [ ] **GUARD-02**: When the webhook endpoint is reachable with no `webhook_secret` set and `webhook_require_secret` off, the operator sees a prominent startup warning that webhooks are unauthenticated — default behavior unchanged, posture no longer silent.
- [ ] **GUARD-03**: When a local delete (`shutil.rmtree`) partially fails, the failure is logged with context rather than silently swallowed — a failed delete leaves an observable signal in the logs (replaces `ignore_errors=True`).
- [ ] **GUARD-04**: The full Python test suite passes under both `fork` and `spawn` start methods — the previously-failing `test_app_process.py::test_process_with_long_running_thread_terminates_properly` now passes, with no test deleted or skipped, because `AppProcess` creates its queue/event from a spawn-compatible multiprocessing context.
- [ ] **GUARD-05**: Tooling/run artifacts (`.orchestrator.json`, `.playwright-mcp/`) are git-ignored so a repo browser never sees stray local artifacts and they cannot be accidentally committed.
- [ ] **GUARD-06**: When startup falls back to the legacy `~/.seedsync` config directory because the configured `--config_dir` is absent, the operator sees a loud one-time warning (or the fallback is gated behind an explicit opt-in) rather than silently loading a pre-fork config.

### Presentation & Launch Readiness (LAUNCH)

Rebuild the public-facing surface so the project's genuine quality is evident in 30 seconds.
Hostile take first (cynical-reader teardown + codex pass per D-5), then the rewrite addresses it.

- [ ] **LAUNCH-01**: A maintainer can read a written "cynical r/selfhosted reader" teardown of the current presentation (README, positioning, first impression), and a codex adversarial pass over the drafted README/docs flags technical-claims accuracy, broken/incomplete install steps, and unsupported assertions — both critiques captured before the rewrite is finalized.
- [ ] **LAUNCH-02**: A visitor reading the README understands what SeedSyncarr is within seconds — a clear one-line description, an above-the-fold screenshot, an honest feature list, accurate install/quickstart instructions, the security posture stated plainly as a selling point, and a short note on the relationship to the original SeedSync fork.
- [ ] **LAUNCH-03**: The README and docs display current, real screenshots of the redesigned UI captured via Playwright against the deployed branch build; any staged state is flagged so nothing misrepresents real behavior.
- [ ] **LAUNCH-04**: The repo includes a `SECURITY.md` with a vulnerability-reporting policy and a short, honest threat-model note (what is protected and what the opt-in security knobs are for).
- [ ] **LAUNCH-05**: The repo includes accurate community-health files — `CONTRIBUTING.md`, issue templates, a PR template, and a correct `LICENSE` — so the absence of these does not read as "not a serious project."
- [ ] **LAUNCH-06**: A clean v1.4.0 release-notes entry exists (so the releases page is presentable), and copy-paste-ready GitHub repo-metadata text (About description, topics/tags, homepage link) is drafted for the maintainer to apply manually.

## Future Requirements

Deferred to a future milestone. Tracked but not in this roadmap.

### Robustness / Tech Debt

- **DEFER-SHUTDOWN**: Replace the fixed `time.sleep` shutdown wait in `seedsyncarr.py` with a deterministic per-job readiness `threading.Event` — real robustness gain, but invisible to a launch reader.
- **DEFER-STREAMQUEUE**: Make `StreamQueue.put` drop-oldest atomic under concurrent producers (wrap get+put in a Lock) — latent, well-mitigated, documented edge-case.
- **DEFER-TESTHARDEN**: Test-suite hardening backlog (`.planning/backlog/TEST-HARDENING-REVIEW.md`, items A-01..A-06 and related) — test-infra quality improvements invisible to a launch reader.
- **DEFER-WEBOB**: Remove the `PYTHONWARNINGS` cgi filter once webob ships a release without the stdlib `cgi` import (upstream PR #466) — blocked on an upstream release; not actionable.

## Out of Scope

Explicitly excluded for v1.4.0. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| New user-facing features | This is a hardening + presentation milestone, not a feature milestone (D-1). |
| Broad refactoring | Only refactoring a *specific finding* justifies is in scope; no opportunistic restructuring. |
| NAS local-build QEMU fix | Deploy-environment limitation, not a code defect; CI multi-arch publish already works. |
| Dual-support GET+POST for config-set | Rejected per D-4 — keeping the insecure GET path alive is exactly what a hostile reader would flag; no external consumers need it. |
| Lidarr/Readarr support | Out of milestone theme; deferred to a future feature milestone. |
| OAuth / multi-user auth | Single-user self-hosted tool. |
