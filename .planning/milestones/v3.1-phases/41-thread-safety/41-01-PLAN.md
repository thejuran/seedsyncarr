Fix two thread-safety issues in controller.py:

**THRD-01: __execute_auto_delete (line ~731)**
The Timer callback runs on a separate thread. Wrap the model access in a `with self.__model_lock:` block. The lock must cover `self.__model.get_file(file_name)` and `self.__file_op_manager.delete_local(file)` together, so no concurrent model update can invalidate the file reference between read and use.

Current code (unsafe):
```python
try:
    file = self.__model.get_file(file_name)
    self.__file_op_manager.delete_local(file)
```

New code:
```python
with self.__model_lock:
    try:
        file = self.__model.get_file(file_name)
    except ModelError:
        self.logger.debug(
            "File '{}' no longer in model, skipping auto-delete".format(file_name)
        )
        return
# delete_local is safe outside lock -- it spawns a subprocess, holding
# the lock during a blocking subprocess call would starve model updates.
# ModelFile is frozen/immutable after add, so `file` reference is safe.
self.__file_op_manager.delete_local(file)
self.logger.info("Auto-deleted local file '{}'".format(file_name))
```

The key insight: `file` is frozen (immutable) after being added to the model, so it is safe to use outside the lock. We only need the lock for the `get_file` call to ensure the file still exists. The `delete_local` call spawns a subprocess and should NOT be under the lock to avoid blocking model updates.

Keep the early-return guards (config disabled, dry-run) before the lock acquisition since they only read config state (not model state).

**THRD-02: __check_webhook_imports (line ~668)**
This method reads model file names, iterates children, and mutates model files -- all without the model lock. Wrap the model-reading section in `with self.__model_lock:`.

Current structure:
1. Build name_to_root dict by iterating model (NEEDS LOCK -- reads model)
2. Call webhook_manager.process() (does NOT need lock -- only reads its own Queue)
3. Update persist and model files for imports (NEEDS LOCK -- mutates model)

Refactor to use two lock windows:

Window 1 (read): Acquire lock, build name_to_root dict, release lock.
Window 2 (mutate): For each newly imported file, acquire lock, read file, create copy, update model, release lock. Then schedule auto-delete outside lock.

The lock windows are kept minimal:
- `webhook_manager.process()` runs outside the lock (it only touches its own thread-safe Queue)
- Auto-delete scheduling runs outside the lock (it only starts a Timer)

Updated method structure:
```python
def __check_webhook_imports(self):
    # Window 1: Build lookup under lock
    name_to_root = {}
    with self.__model_lock:
        for root_name in self.__model.get_file_names():
            name_to_root[root_name.lower()] = root_name
            try:
                root_file = self.__model.get_file(root_name)
                if root_file.is_dir:
                    frontier = list(root_file.get_children())
                    while frontier:
                        child = frontier.pop(0)
                        name_to_root[child.name.lower()] = root_name
                        frontier.extend(child.get_children())
            except ModelError:
                pass

    # Process outside lock (thread-safe Queue)
    newly_imported = self.__webhook_manager.process(name_to_root)

    # Window 2: Update model under lock for each import
    for file_name in newly_imported:
        self.__persist.imported_file_names.add(file_name)
        self.logger.info("Recorded webhook import: '{}'".format(file_name))
        with self.__model_lock:
            try:
                old_file = self.__model.get_file(file_name)
                if old_file.import_status != ModelFile.ImportStatus.IMPORTED:
                    new_file = copy.copy(old_file)
                    new_file._ModelFile__frozen = False
                    new_file.import_status = ModelFile.ImportStatus.IMPORTED
                    self.__model.update_file(new_file)
            except ModelError:
                pass

        # Schedule auto-delete outside lock
        if self.__context.config.autodelete.enabled:
            self.__schedule_auto_delete(file_name)
```

Do NOT change __schedule_auto_delete -- it only touches self.__pending_auto_deletes (a dict accessed only from the controller thread and Timer threads, not the model).
