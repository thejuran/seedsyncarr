  **CRASH-04: Unknown SSE event name crashes subscription**

  In `StreamDispatchService.createSseObserver()` (around line 208-215), the `next` handler in the observable subscription does:
  ```typescript
  this._eventNameToServiceMap.get(eventName).notifyEvent(eventName, eventData);
  ```

  If the server sends an event with a name that was NOT registered (e.g., a new event type added server-side), `.get(eventName)` returns `undefined` and `.notifyEvent()` throws a TypeError. Since this is in the `next` handler of the observable subscription, the error propagates to the observable's `error` handler, which closes the EventSource and reconnects — unnecessarily tearing down a healthy connection.

  Fix: In the `next` handler, check if the event name exists in the map before dispatching:
  ```typescript
  next: (x) => {
      const eventName = x["event"];
      const eventData = x["data"];
      const service = this._eventNameToServiceMap.get(eventName);
      if (service) {
          this._zone.run(() => {
              service.notifyEvent(eventName, eventData);
          });
      } else {
          this._logger.warn("Received unknown SSE event:", eventName);
      }
  },
  ```

  Add a test in stream-service.registry.spec.ts that verifies: when an unknown event name is dispatched via the mock event source, the subscription does NOT error out, and subsequent known events are still delivered correctly. The test pattern:
  1. Open connection (trigger onopen)
  2. Deliver a known event (event1a) — verify it arrives at mockService1
  3. Manually trigger a listener for an unregistered event name (simulate server sending unknown event) — the test should verify no error occurs. Since the EventSource only adds listeners for registered events, this bug manifests when the `observer.next` is called with data whose event name has been removed from the map or was never registered. To simulate this more directly: spy on the `_logger.warn` method and verify it gets called when an unknown event propagates. But the real scenario is that EventSource listeners are only added for known events, so unknown events from the server simply don't trigger any listener. The actual crash scenario is: if the map is modified after initialization or if there's a race condition.

  Actually, the realistic test approach: The EventSource only listens for registered events, so unknown events won't reach the observable. The real danger is if somehow the `_eventNameToServiceMap` changes. A simpler test: directly invoke the subscription's next handler logic by verifying the guard exists. Since the existing test setup fires events through `mockEventSource.listeners.get("event1a")`, we can verify the fix by adding a test that confirms the dispatch service does not crash when the internal map is queried for an unknown event. The simplest test: verify the observable subscription continues working after processing events (it doesn't error on known events). The guard is a defensive measure.

  For a proper test: after the fix, trigger a known event, then check the service still works. The guard is testable by verifying the `_logger.warn` spy is NOT called for known events, implying the guard path exists.

  **CRASH-05: Malformed JSON in SSE message crashes observable chain**

  All three stream services call `JSON.parse(data)` without try/catch. If the server sends a malformed JSON payload (e.g., truncated due to network issues, encoding error, or server bug), JSON.parse throws a SyntaxError. This exception propagates up through `notifyEvent` → the observable's `next` handler, which turns it into an observable error, tearing down the SSE connection.

  **Fix model-file.service.ts `parseEvent` method:**
  Wrap the entire body of `parseEvent` in a try/catch. On catch, log the error with `this._logger.error("Failed to parse SSE event:", name, error)` and return early. This way a single bad event is logged and skipped, and the observable chain continues processing subsequent events.

  ```typescript
  private parseEvent(name: string, data: string): void {
      try {
          // ... existing if/else chain for EVENT_INIT, EVENT_ADDED, etc.
      } catch (error) {
          this._logger.error("Failed to parse model event:", name, error);
      }
  }
  ```

  **Fix server-status.service.ts `parseStatus` method:**
  Wrap `parseStatus` body in try/catch:
  ```typescript
  private parseStatus(data: string): void {
      try {
          const statusJson: ServerStatusJson = JSON.parse(data);
          const status = ServerStatus.fromJson(statusJson);
          this._status.next(status);
      } catch (error) {
          this._logger.error("Failed to parse status event:", error);
      }
  }
  ```
  Note: ServerStatusService constructor doesn't inject LoggerService. Either add it as a constructor dependency, or use `console.error` as a fallback. Check the existing constructor — if no _logger exists, inject LoggerService the same way ModelFileService does.

  **Fix log.service.ts `onEvent` method:**
  Wrap the JSON.parse call in try/catch:
  ```typescript
  protected onEvent(eventName: string, data: string): void {
      try {
          this._hasReceivedLogs = true;
          this._logs.next(LogRecord.fromJson(JSON.parse(data)));
      } catch (error) {
          console.error("Failed to parse log event:", error);
      }
  }
  ```
  Note: LogService also doesn't inject LoggerService. Use `console.error` here to avoid adding a new dependency for a defensive error path. Or optionally inject LoggerService if it's already available in the module.

  **Tests:**
  Add a test in each service's spec file that calls `notifyEvent` with invalid JSON (e.g., `"not valid json {{{"`), and verifies:
  1. No exception is thrown
  2. The service's observable state is unchanged (e.g., files map stays the same, status stays the same, no new log record emitted)
  3. Subsequent valid events still work correctly
