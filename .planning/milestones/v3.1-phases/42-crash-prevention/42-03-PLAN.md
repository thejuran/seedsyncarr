  **CRASH-06: Individual action endpoints wait indefinitely**

  The 5 individual action handler methods (`__handle_action_queue`, `__handle_action_stop`, `__handle_action_extract`, `__handle_action_delete_local`, `__handle_action_delete_remote`) all call `callback.wait()` with no timeout argument (line 85, 104, 123, 142, 161). This means if the controller never processes the command (e.g., it's stuck, crashed, or overwhelmed), the web server thread blocks forever.

  Note: The bulk endpoint already uses bounded timeouts (`callback.wait(timeout=remaining_timeout)`). The individual endpoints need the same treatment.

  Fix:
  1. Add a class-level constant for the individual action timeout:
     ```python
     # Timeout for individual action endpoints in seconds
     _ACTION_TIMEOUT = 30.0
     ```

  2. Update all 5 `callback.wait()` calls to use the timeout:
     ```python
     completed = callback.wait(timeout=self._ACTION_TIMEOUT)
     ```

  3. Add timeout handling after each wait. If `completed` is False, return a 504 response:
     ```python
     completed = callback.wait(timeout=self._ACTION_TIMEOUT)
     if not completed:
         return HTTPResponse(body="Operation timed out", status=504)
     if callback.success:
         return HTTPResponse(body="Queued file '{}'".format(file_name))
     else:
         return HTTPResponse(body=callback.error, status=callback.error_code)
     ```

  Apply this pattern to all 5 handlers. The timeout value of 30 seconds is appropriate: normal commands complete in milliseconds, but delete/extract operations can take seconds; 30s provides generous headroom while still bounding the wait.

  **Tests:**
  Add tests in test_controller_handler.py:

  1. `test_action_timeout_returns_504`: Mock the controller's `queue_command` to NOT call any callback (simulating a stuck controller). Call one of the action handlers (e.g., queue). Since the callback.wait() will timeout, verify the response has status 504 and body contains "timed out". To make this test fast, temporarily set `_ACTION_TIMEOUT` to a small value (e.g., 0.1 seconds) on the handler instance before calling, or mock the wait method.

  2. `test_action_success_returns_200`: Mock the controller to call `on_success()` on the callback synchronously. Call an action handler and verify 200 response. (This may already be implicitly tested, but an explicit test confirms the timeout path doesn't break the success path.)

  For the timeout test, the simplest approach: patch `ControllerHandler._ACTION_TIMEOUT` to 0.1 in the test to avoid waiting 30 seconds. Or mock the callback's wait method to return False.
