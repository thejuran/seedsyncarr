  <name>Task 1: Fix AppComponent router subscription leaks</name>
  <files>

  <action>
AppComponent has THREE subscription leaks:
1. `router.events.subscribe(...)` in the constructor (line 34) — not stored, not unsubscribed
2. `this.router.events.subscribe(...)` in ngOnInit (line 42) — not stored, not unsubscribed
3. `this._toastSubscription` is already properly managed (stored + unsubscribed in ngOnDestroy) — leave it

Fix approach using the takeUntil/destroy$ pattern (consistent with existing codebase patterns):

1. Add `Subject` import (already has `Subscription` from rxjs)
2. Add `takeUntil` import from `rxjs/operators`
3. Add a private `destroy$ = new Subject<void>()` field
4. Move the constructor's router.events subscription into ngOnInit (constructor should not have side effects with subscriptions)
5. Add `.pipe(takeUntil(this.destroy$))` to BOTH router.events subscriptions
6. Update ngOnDestroy to call `this.destroy$.next()` and `this.destroy$.complete()` (keep the existing _toastSubscription cleanup)
7. Remove the `_toastSubscription` field and its Subscription import IF you convert it to takeUntil as well — OR keep it as-is (it already works). Prefer consistency: convert it to takeUntil for uniformity.

After refactoring, ngOnDestroy should be:
```typescript
ngOnDestroy(): void {
    this.destroy$.next();
    this.destroy$.complete();
}
```

The two router subscriptions should have `.pipe(takeUntil(this.destroy$))` before `.subscribe(...)`.
The toast subscription should also use `.pipe(takeUntil(this.destroy$))` for consistency, removing the manual _toastSubscription field.
