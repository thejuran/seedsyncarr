<objective>
Add a `StorageStatus` component to the backend `Status` model with four nullable byte-count properties (`local_total`, `local_used`, `remote_total`, `remote_used`), and extend `SerializeStatusJson.status()` to emit those fields as a top-level `"storage"` block on the existing SSE status stream. This is the foundation that Plan 02 (scanners wire real values into) and Plan 03 (frontend consumes from).

Purpose: The design spec (locked in `docs/superpowers/specs/2026-04-15-storage-capacity-tiles-design.md`) mandates piggybacking capacity data onto the existing `Status` SSE stream (Approach A) with no new endpoint. This plan delivers the model + serializer end of that piggyback. Per D-14/D-15, all four fields default to `None` so the frontend's cold-load path is "tracked-bytes fallback until a non-null (total, used) pair arrives."

Output: Extended `status.py` + `serialize_status.py` + unit tests. No behavior change yet — values stay `None` until Plan 02 populates them.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Status listener → SSE stream | `StatusStreamHandler` consumes `Status` property changes and broadcasts serialized JSON to all authenticated dashboard viewers |
| JSON serialization → dashboard client | Serialized payload crosses the web boundary to any user who can reach the status SSE endpoint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-74-01 | Information Disclosure | `SerializeStatusJson.status()` emitting `storage` block | accept | Capacity numbers (total/used bytes) reveal disk sizing to authenticated dashboard viewers. This is the same exposure class as existing tracked-bytes and scan-time fields. No new disclosure class, no PII. Existing auth gate on `/status` SSE endpoint (Phase 50 token auth) remains the sole control. |
| T-74-02 | Tampering | `Status.StorageStatus` properties written from outside the scanner | accept | `Status` model is process-internal; only the controller process mutates it. The `Cannot reassign component` guard (tested in Task 1) prevents wholesale component replacement. Individual property writes are by design — that's how the scanner updates state. |
| T-74-03 | Denial of Service | Fresh-status default-None values causing client-side crashes | mitigate | All four fields default to `None` and serialize as JSON `null`. Frontend (Plan 03) handles null explicitly via per-tile `@if` gates. Unit tests assert the null-default behavior to catch regressions. |
| T-74-04 | Injection | New JSON keys colliding with user-controlled input | accept | Keys are fixed string constants (`"storage"`, `"local_total"`, etc.) hard-coded in `SerializeStatusJson`. Values are numeric or None — no string interpolation from user input into the serialized payload. XSS surface is unchanged from the existing controller/server blocks. |
</threat_model>
