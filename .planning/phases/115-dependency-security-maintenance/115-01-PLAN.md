## The SHA-pinned merge gate (apply to every merge, no exceptions)

For a target PR <N>, perform these steps in order. Merge ONLY if every step passes for the SAME head SHA.

1. CAPTURE the current head SHA and mergeability in one read:
   `gh pr view <N> --json state,mergeable,headRefOid,headRefName`
   Record `headRefOid` as SHA_PINNED. The precondition is `state == OPEN` AND `mergeable == MERGEABLE`.
   - `mergeable == MERGEABLE` is the GitHub gate (D-02). Do NOT additionally require mergeStateStatus == CLEAN, and do NOT block merely because the PR is BEHIND main — BEHIND alone is fine if GitHub still reports MERGEABLE and CI is green for the current head (next step).
   - If `mergeable` is CONFLICTING/UNKNOWN (not MERGEABLE), treat as not-mergeable → go to the bounded-wait path below.

2. VERIFY CI is green for SHA_PINNED:
   `gh pr checks <N>` — every blocking gate ("Lint (Python)", "Lint (Angular)", "Python unit tests", "Angular unit tests", "Build Docker Image", "End-to-end tests on Docker Image (amd64)", "Release metadata verifier tests", "Verify release metadata") must report pass/SUCCESS. Publish/CodeQL jobs reporting skipping/SKIPPED/NEUTRAL is expected and is NOT a red gate.
   `gh pr checks` reports checks for the PR's current head; treat its result as the verdict for SHA_PINNED. If any blocking gate is pending/failing/stale → go to the bounded-wait path below.

3. MERGE pinned to SHA_PINNED (the race-proof step):
   `gh pr merge <N> --squash --delete-branch --match-head-commit SHA_PINNED`
   This makes GitHub REFUSE the merge if the head moved (Dependabot rebased/pushed) between steps 1-2 and now. If it refuses because the head moved, that is the gate working correctly — return to step 1 and re-run the whole gate against the NEW head SHA.
   - FALLBACK (only if the installed `gh` lacks `--match-head-commit`; the planning-time gh 2.86.0 has it): immediately before merging, re-read `gh pr view <N> --json headRefOid`. If it differs from SHA_PINNED, ABORT the merge and return to step 1 for the new SHA. Only if it still equals SHA_PINNED, run `gh pr merge <N> --squash --delete-branch`.

4. CONFIRM the merge landed: `gh pr view <N> --json state --jq .state` returns MERGED.

## Bounded-wait path (only for not-green or not-mergeable — NOT for BEHIND)

Trigger this path ONLY when step 1 reports not-mergeable, or step 2 reports a blocking gate pending/failing/stale. Do NOT trigger it merely because a PR is BEHIND main.

- If the cause is staleness against main and you want main-freshness explicitly, take an EXPLICIT action rather than passively waiting on Dependabot: run `gh pr update-branch <N>` to rebase the PR onto current main, then return to step 1 (a new head SHA will be produced; re-pin and re-check).
- Otherwise (CI genuinely pending/running after a Dependabot rebase), poll `gh pr view <N> --json mergeable,headRefOid` + `gh pr checks <N>` periodically until checks are green for the current head, then re-enter the gate at step 1 with the fresh SHA.
- BOUND: if the PR cannot reach "MERGEABLE + green for its current head" within ~20-30 min of polling/one update-branch attempt, STOP. Leave it and the remaining PRs unmerged. Surface the stuck PR (number, last `mergeable`, last `headRefOid`, failing/pending gate) to the operator in the SUMMARY. Never `--admin`/force past a red gate.

## Replacement-PR rule (a closed/superseded target is a BLOCKER, not a silent drop)

If a target bump #60-#66 is NOT open at merge time (already closed/superseded by Dependabot):
- It is a BLOCKER unless a replacement Dependabot PR for the SAME underlying bump is explicitly identified (e.g. Dependabot opened a newer PR superseding it). Find it via `gh pr list --state open --json number,title,headRefName` and match the package/bump.
- If identified: insert the replacement into the ordered merge sequence in the SAME risk slot as the original (a hono replacement goes in #65's slot, a Python dev-dep replacement in its #60-#63 slot), merge it through this identical SHA-pinned gate, and RECORD "original #<X> superseded by #<Y>; #<Y> merged satisfies that bump" for the SUMMARY and the Task 4 accounting.
- If NO replacement can be identified: HALT. Do not silently continue with a smaller set. Surface the orphaned bump (original PR number, its closed state, the package it was bumping, and the still-open alert if any) to the operator in the SUMMARY.
