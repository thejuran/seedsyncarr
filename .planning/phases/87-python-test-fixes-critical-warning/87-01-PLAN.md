**PYFIX-01** (per D-01 -- fix in place): In `test_extract_process.py` at line 182, change:
```python
threading.Thread(target=_callback_sequence()).start()
```
to:
```python
threading.Thread(target=_callback_sequence).start()
```
Remove the `()` after `_callback_sequence` so it is passed as a callable to `Thread`, not invoked immediately. The `()` causes `_callback_sequence` to run synchronously in the calling thread and passes its return value (`None`) as the thread target -- so the background thread never runs the body.

After fix, the existing `while self.completed_signal.value < 1: pass` busy-wait loops at lines 188 and 198 will correctly gate on the background thread completing.

**PYFIX-02** (per D-01 -- fix in place): In `test_lftp_manager.py`, find the method `test_init_skips_rate_limit_when_zero` (around line 83). Replace the `pass` statement on line 98 with an explicit assertion:
```python
self.assertNotEqual(mock_lftp.rate_limit, 0)
```
Remove the comment `# No assertion needed -- test passes if __init__ doesn't crash` and replace with:
```python
# MagicMock auto-creates attributes on access as MagicMock objects (not 0).
# Production code only assigns rate_limit when rate_limit > 0.
# If rate_limit were assigned 0, this assertion would catch it.
self.assertNotEqual(mock_lftp.rate_limit, 0)
```

This works because `MagicMock` auto-attributes are `MagicMock` objects, never equal to `0`. The production code at `lftp_manager.py:57-58` assigns `self.__lftp.rate_limit = rate_limit` only when `rate_limit > 0`. When 0, the attribute is never assigned, so `mock_lftp.rate_limit` remains the auto-created `MagicMock` object.
