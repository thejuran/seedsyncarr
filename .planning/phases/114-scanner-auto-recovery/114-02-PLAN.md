<objective>
Add a bounded controller auto-restart safety net (RECOV-01): a permanent-class controller death routes into the EXISTING `ServiceRestart` → `main()` `continue` machinery instead of staying down, capped at a fixed number of consecutive AUTO restarts with a stayed-up reset that is evaluated AT FAILURE TIME using the current run's age, falling through to today's `server.up=False` + `error_msg` surface once the cap is exhausted. UI-requested restarts must not burn the auto budget. This is the controller-recovery half of Phase 114, scoped to `seedsyncarr.py` only.

Purpose: In the 2026-06-19 incident the controller died on a permanent-class error and the web server stayed up, so the UI looked fine while the file list froze for ~2 days. Routing the first deaths through the proven restart path recovers automatically; bounding it ensures a genuinely unrecoverable condition still surfaces (and does not become a restart loop). The decision MUST be made at failure time using the current run's elapsed time (not a stale bool precomputed before app.run()), so a run that has stayed healthy past the reset threshold is recognized as restart-eligible even when the consecutive counter was already at the cap (finding 1). And the auto-recovery budget must be isolated from UI-requested restarts so a manual restart never exhausts it (finding 1).
Output: A pure, unit-testable `_should_auto_restart(...)` static helper that computes the stayed-up reset from the current run's age, restart-cap/reset constants local to `seedsyncarr.py`, the `run()` `AppError`-catch and `main()` restart-loop wiring with restart-source tracking, and unit tests proving within-cap / cap-exhausted / reset-at-cap / UI-does-not-burn-budget behavior.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| controller child process → supervisor (seedsyncarr.py run/main) | A permanent-class error propagates up as AppError; the supervisor decides restart-vs-surface at failure time |
| UI restart request → restart machinery | A user-initiated restart shares the ServiceRestart signal; it must be distinguishable from an auto-recovery restart so it does not consume the auto budget |
| restart machinery → system resources | Repeated auto restarts (terminate/join jobs, re-init) consume CPU/IO; an unbounded loop would be a self-inflicted resource drain |
