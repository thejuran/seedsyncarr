<objective>
Add a bounded controller auto-restart safety net (RECOV-01): a permanent-class controller death routes into the EXISTING `ServiceRestart` → `main()` `continue` machinery instead of staying down, capped at a fixed number of consecutive restarts with a stayed-up reset, falling through to today's `server.up=False` + `error_msg` surface once the cap is exhausted. This is the controller-recovery half of Phase 114, scoped to `seedsyncarr.py` only.

Purpose: In the 2026-06-19 incident the controller died on a permanent-class error and the web server stayed up, so the UI looked fine while the file list froze for ~2 days. Routing the first deaths through the proven restart path recovers automatically; bounding it ensures a genuinely unrecoverable condition still surfaces (and does not become a restart loop).
Output: A pure, unit-testable `_should_auto_restart(...)` static helper, restart-cap/reset constants local to `seedsyncarr.py`, the `run()` `AppError`-catch and `main()` restart-loop wiring, and unit tests proving within-cap / cap-exhausted / reset behavior.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| controller child process → supervisor (seedsyncarr.py run/main) | A permanent-class error propagates up as AppError; the supervisor decides restart-vs-surface |
| restart machinery → system resources | Repeated restarts (terminate/join jobs, re-init) consume CPU/IO; an unbounded loop would be a self-inflicted resource drain |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-114-05 | Denial of Service | Restart loop becoming a resource-exhaustion vector (tight crash-restart-crash churn) | mitigate | Bounded consecutive-restart cap (`__RESTART_CAP = 3`) + stayed-up reset (`__RESTART_RESET_SECS = 300`); after the cap, fall through to today's server.up=False surface — never `raise ServiceRestart()` unconditionally (RECOV-01 / D-03). |
| T-114-06 | Spoofing | Auto-restart masking a genuinely unrecoverable condition (controller never able to come up) | mitigate | The cap guarantees the failure surfaces after a bounded number of restarts via the unchanged server.up=False + error_msg path; the reset only fires when a run actually stayed up past the threshold, so a persistently-failing controller cannot reset its budget. |
| T-114-07 | Repudiation | Restart bookkeeping log lines | mitigate | Lazy `%`-style logging of restart counters only (no secrets, no untrusted strings); consistent with the existing "Restarting..." log at main(). |
| T-114-SC | Tampering | npm/pip/cargo installs | accept | No packages installed in this phase (stdlib only: datetime). RESEARCH "Package Legitimacy Audit: Not applicable — zero external packages." No legitimacy checkpoint required. |
