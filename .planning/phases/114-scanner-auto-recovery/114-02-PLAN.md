<objective>
Add a bounded controller auto-restart safety net (RECOV-01): a permanent-class controller death routes into the EXISTING `ServiceRestart` → `main()` `continue` machinery instead of staying down, capped at a fixed number of consecutive AUTO restarts with a stayed-up reset that is evaluated AT FAILURE TIME using the current run's age, falling through to today's `server.up=False` + `error_msg` surface once the cap is exhausted. The stayed-up reset must produce a FRESH budget (counter normalized to 1), not a one-off pass that leaves the counter at cap+1. UI-requested restarts must not burn the auto budget. This is the controller-recovery half of Phase 114, scoped to `seedsyncarr.py` + the `ServiceRestart` flags in `common/error.py`.

Purpose: In the 2026-06-19 incident the controller died on a permanent-class error and the web server stayed up, so the UI looked fine while the file list froze for ~2 days. Routing the first deaths through the proven restart path recovers automatically; bounding it ensures a genuinely unrecoverable condition still surfaces (and does not become a restart loop). The decision MUST be made at failure time using the current run's elapsed time (not a stale bool precomputed before app.run()), so a run that has stayed healthy past the reset threshold is recognized as restart-eligible even when the consecutive counter was already at the cap (finding 1). Critically, that recognition must REFRESH the budget: when a stayed-up run dies AT the cap, the very next quick failure must still be allowed (counter starts at 1, not cap+1), otherwise the cap-exhaustion bug persists for the multi-failure case (finding 2 — the codex follow-on). And the auto-recovery budget must be isolated from UI-requested restarts so a manual restart never exhausts it (finding 1). The `ServiceRestart` flag addition must NOT regress the inherited Exception message contract: `ServiceRestart("restart requested")` must still preserve that message via `str(e)` (there is a live test asserting exactly this), and the flags must be keyword-only so a positional message can never be mistaken for the `auto` flag (codex HIGH finding).
Output: A pure, unit-testable `_should_auto_restart(...) -> (should_restart, reset_applied)` static helper that computes the stayed-up reset from the current run's age, restart-cap/reset constants local to `seedsyncarr.py`, keyword-only `auto`/`reset` flags on `ServiceRestart` (added via an `__init__(self, *args, auto=False, reset=False)` that passes `*args` through to `super().__init__(*args)` so the message contract is preserved), the `run()` `AppError`-catch and `main()` restart-loop wiring with restart-source tracking and counter normalization, and unit tests proving within-cap / cap-exhausted / reset-at-cap / fresh-budget-after-reset / UI-does-not-burn-budget behavior plus the ServiceRestart constructor compatibility cases.
</objective>
