<objective>
Reclassify transient name-resolution failures as in-scan-retryable and add a bounded retry loop that retries ONLY name-resolution failures so the seedbox scanner survives a transient DNS blip (SCAN-01, SCAN-02), while preserving the exact existing failure surface once name-resolution retries are exhausted AND leaving every other error class on its current path unchanged (SCAN-03). This is the recovery half of Phase 114, scoped to the SSH scan path only.

Purpose: A transient DNS failure resolving the seedbox host currently raises `SshcpError("Bad hostname: {host}")`, is classified non-recoverable, kills the scanner, and freezes the file list for days. Name-resolution failures fail fast (no multi-second SSH wait), so retrying them in-scan within a tight bounded window lets a DNS blip clear without the scanner dying, while a genuinely wrong hostname still stops after the bound. Timeout / connection-refused errors are deliberately NOT retried in-scan — they can each block for up to the Sshcp 180s command timeout, so retrying them in-scan could stall the scanner for minutes; they keep today's behavior (raise, recover on the next ~30s scan interval).
Output: A new `NAME_RESOLUTION_ERROR_PATTERNS` tuple + matcher, a bounded retry loop wrapping only the main scan SSH call inside `RemoteScanner.scan()` whose retry condition is name-resolution ONLY, a `__to_scanner_error` helper preserving the __first_run-aware immediate-raise for everything else, and unit tests proving recover/exhaust/bounded/classification behavior plus that transient errors are not retried in-scan.
</objective>
