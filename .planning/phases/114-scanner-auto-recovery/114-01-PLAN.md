<objective>
Reclassify transient name-resolution failures as in-scan-retryable and add ONE bounded retry implementation that retries ONLY name-resolution failures, wrapping BOTH the main scan SSH call AND the install-path SSH operations, so the seedbox scanner survives a transient DNS blip wherever it strikes (SCAN-01, SCAN-02) — including at startup and after every auto-restart, when the install path runs and __install_done is false. The exact existing failure surface is preserved once name-resolution retries are exhausted, AND every other error class stays on its current immediate-raise path unchanged (SCAN-03). This is the recovery half of Phase 114, scoped to the SSH scan + install paths.

Purpose: A transient DNS failure resolving the seedbox host currently raises a name-resolution `SshcpError`, is classified non-recoverable, kills the scanner, and freezes the file list for days. CRITICAL coverage gap (codex HIGH): `scan()` calls `_install_scanfs()` BEFORE the main scan call whenever `__install_done` is false — which is exactly true at startup and after every auto-restart. `_install_scanfs` converts a name-resolution `SshcpError` to `ScannerError(recoverable=self._is_transient_ssh_error(e))` → `recoverable=False` for a name-resolution string → a startup/post-restart DNS blip bypasses the name-resolution retry loop, kills the controller, AND burns RECOV-01 restart budget, defeating the milestone. SECOND CRITICAL gap (codex HIGH, this rewrite): the name-resolution matcher must cover ALL resolver-string surfaces the SSH layer can actually present — not only the collapsed `Bad hostname:`. The two literally-matched pexpect expect indices (3 `Could not resolve hostname` / 5 `Name or service not known`) collapse to `Bad hostname: {host}` at sshcp.py:97-98 and :128-129, BUT the non-zero-exit fallthrough at sshcp.py:155 raises the RAW `sp.before` string, so a resolver output like `ssh: Could not resolve hostname moon.usbx.me: Temporary failure in name resolution` reaches RemoteScanner as raw text, NOT collapsed. The existing SSH contract proves this: test_sshcp.py:108-115 and :172-178 accept bad-host errors containing `bad hostname`, `name or service not known`, `could not resolve`, and `temporary failure` (lower-cased). If the matcher only knew `Bad hostname:`, those real resolver strings would be non-retryable on BOTH the main scan and the install path — a startup/post-restart DNS blip on those platforms would still raise immediately as fatal, kill the controller, and burn RECOV-01 budget despite the shared helper. Name-resolution failures fail fast (no multi-second SSH wait), so retrying them in-scan within a tight bounded window lets a DNS blip clear without the scanner dying — on EITHER path — while a genuinely wrong hostname still stops after the bound. Timeout / connection-refused errors are deliberately NOT retried in-scan on either path — they can each block for up to the Sshcp 180s command timeout, so retrying them in-scan could stall the scanner for minutes; they keep today's behavior (immediate raise, recover on the next ~30s scan interval or — for install — the next scan tick).
Output: A new `NAME_RESOLUTION_ERROR_PATTERNS` tuple (covering the collapsed AND raw resolver surfaces) + a case-insensitive matcher, ONE shared bounded retry helper inside `RemoteScanner` whose retry condition is name-resolution ONLY and which wraps both the main scan SSH call and the install-path md5sum/copy SSH calls, a `__to_scanner_error` helper preserving the __first_run-aware immediate-raise for everything else on the main-scan path, and unit tests proving recover/exhaust/bounded/classification behavior — including for the RAW resolver strings — on BOTH paths plus that transient errors are not retried in-scan on either path.
</objective>
