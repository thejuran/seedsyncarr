### `src/docker/test/python/Dockerfile` — TECH-01

**Role:** Dockerfile (test image), modify `RUN apt-get install` block
**Analog:** `src/docker/test/angular/Dockerfile:6-17` — same repo, same arch-gating problem (Chrome not on arm64), same `apt-get install` context.

**Analog — `src/docker/test/angular/Dockerfile:6-17`**:
```dockerfile
# Install Chrome/Chromium for headless testing
# ARM64: Use Chromium from Debian repos (Chrome not available for ARM64 Linux)
# AMD64: Use Google Chrome
RUN apt-get update && \
    ARCH=$(dpkg --print-architecture) && \
    if [ "$ARCH" = "arm64" ]; then \
        apt-get install -y chromium chromium-driver && \
        ln -sf /usr/bin/chromium /usr/bin/google-chrome; \
    else \
        apt-get install -y wget gnupg && \
        ...snip... \
        apt-get install -y google-chrome-stable; \
    fi
```

**What to replicate:**
- Shell-side arch check inside a single `RUN` layer (one `apt-get update` shared for amd64 + arm64).
- `if [ "$ARCH" = "arm64" ]; then … else … fi` structure with `&& \` line continuations.
- Comment block above the `RUN` naming both architectures and why they differ.
- One shared apt packages list (`openssh-server unrar`) installed unconditionally; the arch-specific package (`rar`) gated inside the `if`.

**Discretionary choice for planner — two viable idioms:**

1. **`dpkg --print-architecture`** (exact match to existing `src/docker/test/angular/Dockerfile` analog above). Works under plain `docker build` and under `docker compose build` uniformly — no reliance on buildx arg injection. This is the most consistent choice with current repo convention.

2. **`ARG TARGETARCH`** (RESEARCH.md §4.6 recommendation). Buildx-native; requires the `|| [ -z "$TARGETARCH" ]` empty-check to preserve CI amd64 path when buildx isn't driving (RESEARCH.md §8.2). Not currently used anywhere in this repo.

**Recommendation for planner:** Use the `dpkg --print-architecture` idiom. It matches the existing analog in `src/docker/test/angular/Dockerfile` exactly, avoids the empty-`TARGETARCH` pitfall, and one fewer deviation from established repo convention. If the planner insists on `ARG TARGETARCH` per RESEARCH.md §4.6, the empty-case fallback (`|| [ -z "$TARGETARCH" ]`) is mandatory — see RESEARCH.md §8.2.

**Exact file excerpt to modify** (`src/docker/test/python/Dockerfile:5-11`):
```dockerfile
# Install dependencies
# Note: The base image (seedsyncarr_run_python_env) already has non-free repos enabled via DEB822 format
RUN apt-get update && \
    apt-get install -y \
        openssh-server \
        rar \
        unrar
```

**Target shape (dpkg idiom, matching angular analog)**:
```dockerfile
# Install dependencies
# Note: The base image (seedsyncarr_run_python_env) already has non-free repos enabled via DEB822 format
# ARM64: skip `rar` (Debian ships rar for amd64/i386 only; RARLAB has no aarch64 Linux build)
# AMD64: install `rar` normally for test-fixture archive creation
RUN apt-get update && \
    apt-get install -y openssh-server unrar && \
    ARCH=$(dpkg --print-architecture) && \
    if [ "$ARCH" = "amd64" ] || [ "$ARCH" = "i386" ]; then \
        apt-get install -y rar; \
    else \
        echo "Skipping rar install on $ARCH (Debian rar is amd64/i386 only)"; \
    fi
```
