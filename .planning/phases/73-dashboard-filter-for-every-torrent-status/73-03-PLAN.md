<objective>
Add URL query-parameter persistence to `TransferTableComponent`. On init, read `?segment=` and `?sub=` from `ActivatedRoute.snapshot.queryParamMap`, validate against the enum (silent fallback to `segment=all` for invalid values per D-11), and hydrate `activeSegment` + `activeSubStatus` BEFORE the first `filterState$.next()` is observed by template subscribers. On every `onSegmentChange` / `onSubStatusChange`, write the new state to the URL via `Router.navigate([], { relativeTo: this.route, queryParams, queryParamsHandling: 'merge' })` so other unrelated query params remain untouched.

This is a NEW pattern in the codebase — no existing component reads or writes URL query params via `ActivatedRoute.queryParamMap` or `Router.navigate`. The implementation follows CONTEXT D-09/D-10/D-11 + Angular's documented `NavigationExtras.queryParamsHandling` contract.

Purpose: D-09 (persist filter via URL query params; omit `sub` when no sub-status; omit both when `segment === 'all'`), D-10 (use `Router.navigate` with `queryParamsHandling: 'merge'`), D-11 (silent fallback to `segment=all` for invalid values; no error toast; page state and `nameFilter` are NOT persisted).

Output: `transfer-table.component.ts` gains `Router` and `ActivatedRoute` injection, an `ngOnInit` that hydrates state from the URL, a private `_writeFilterToUrl()` helper, and call sites at the end of `onSegmentChange` and `onSubStatusChange` that invoke the helper.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL → component state | An attacker-controlled URL (`?segment=...&sub=...`) is read by `ngOnInit` and feeds `activeSegment` / `activeSubStatus`. This is the ONE new attack surface in Phase 73. |
| component state → URL | `Router.navigate` writes `queryParams` derived from internal `activeSegment` / `activeSubStatus` values. Both are typed and bounded; this direction is low-risk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-73-03-01 | Tampering | URL `?segment` value | mitigate | Validated against the literal allow-list `VALID_SEGMENTS = {"all","active","done","errors"}` in `ngOnInit`. Any other value silently coerces to `"all"` per D-11. The validated value is then assigned to `activeSegment` (typed string union). No raw string is ever assigned. |
| T-73-03-02 | Tampering | URL `?sub` value | mitigate | Validated against the per-segment allow-list `VALID_SUBS_PER_SEGMENT[segment]` (a `Set<string>` of literal `ViewFile.Status` enum strings). Any other value silently coerces to `null`. A sub that doesn't belong to the named segment also silently coerces to `null` (e.g., `?segment=active&sub=stopped` resolves to active+null). |
| T-73-03-03 | Open redirect / path manipulation | `Router.navigate` call | mitigate | Navigate is invoked with commands array `[]` AND `relativeTo: this.route` — only the current route's query params can be modified. No path component, no fragment, no cross-route navigation is possible from any query-param value. |
| T-73-03-04 | XSS via reflected query param | template rendering of activeSegment / activeSubStatus | accept | The template (Plan 02) compares these values against literal enum strings in `[class.active]` and `(click)` bindings. Neither is interpolated into text content via `{{ }}` or `[innerHTML]`. Even if a validation bypass existed, the reflected value would never reach the DOM as raw HTML. |
| T-73-03-05 | Information Disclosure | URL leakage of filter state | accept | The URL reveals which filter is active (e.g., `?segment=errors&sub=stopped` reveals the user is viewing failed transfers). No PII, no credentials, no file names. Acceptable for a self-hosted single-user app. |
| T-73-03-06 | Denial of Service | repeated `Router.navigate` calls causing reflow | accept | Each filter click invokes `navigate` exactly once. `replaceUrl: true` avoids history bloat. Angular Router debounces internal state changes; no unbounded loop exists between `filterState$.next()` and `_writeFilterToUrl()` because the helper does not subscribe to `filterState$`. |
