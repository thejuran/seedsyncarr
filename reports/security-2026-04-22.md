## Raw Tool Output & Triage

### Semgrep — 630 raw detections, 0 true positives

All 630 detections were triaged as **false positives** caused by overly broad Semgrep rules:

#### `js-nosql-injection-where` — 617 false positives

The rule pattern `$COLLECTION.$WHERE($INPUT)` matches any method call on any object. It fired on standard Angular/TypeScript code like `this.cdr.markForCheck()`, `service.subscribe(...)`, `array.push(...)`, etc. None involve MongoDB's `$where` operator.

**Recommendation:** This rule needs a `pattern-inside` constraint to limit matches to MongoDB collection contexts, or should be scoped to files importing a MongoDB driver.

#### `js-xss-eval-user-input` — 11 false positives

The rule matches `setTimeout($INPUT, ...)` and `setInterval($INPUT, ...)`. All 11 hits pass arrow-function callbacks (e.g., `setTimeout(() => this.scrollToBottom(), 0)`), not string arguments. Only string arguments to setTimeout/setInterval would enable code injection.

Files affected:
- `logs-page.component.ts` (lines 87, 175)
- `app.component.ts` (line 56)
- `settings-page.component.ts` (lines 194, 205, 218)
- `stream-service.registry.ts` (lines 116, 163, 259)
- `bulk-action-dispatcher.service.ts` (line 162)
- `confirm-modal.service.ts` (line 167)

**Recommendation:** Add `pattern-not: setTimeout((...) => {...}, ...)` and `pattern-not: setInterval((...) => {...}, ...)` to the rule to exclude arrow function callbacks.

#### `js-prototype-pollution-merge` — 1 false positive

- **File:** `cached-reuse-strategy.ts:19`
- **Code:** `this.handlers[route.routeConfig.path] = handle`
- **Verdict:** Key comes from Angular's compile-time route config, not user input. The `handlers` object is a plain cache for Angular's `RouteReuseStrategy`.

#### `js-xss-innerhtml` — 1 false positive

- **File:** `confirm-modal.service.ts:100`
- **Code:** `this.modalElement!.innerHTML = \`...\``
- **Verdict:** All interpolated values are escaped through `ConfirmModalService.escapeHtml()` (lines 33-40, applied at lines 51-56). The `skipMessage` only interpolates an integer count. No user-controlled data reaches innerHTML unescaped.

## Recommendations

1. **Tighten Semgrep rule `js-nosql-injection-where`** — The `$COLLECTION.$WHERE($INPUT)` pattern is too broad and generates hundreds of false positives on any TypeScript codebase. Scope it to MongoDB contexts or replace with a more precise pattern.

2. **Tighten Semgrep rule `js-xss-eval-user-input`** — Add exclusions for arrow-function and named-function callbacks to `setTimeout`/`setInterval` patterns. Only string arguments represent an actual eval sink.

3. **Consider adding Semgrep to CI** — Now that Semgrep is on PATH, it can run in CI. The rule fixes above should be applied first to avoid noisy pipelines.

4. **Run `pip-audit`** periodically — Python dependencies were not scanned this round since no changes were detected, but should be checked on a regular cadence.
